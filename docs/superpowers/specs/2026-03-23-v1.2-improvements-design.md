## 1. Multi-Terminal Awareness

### Problem

Each Claude Code terminal tracks its own session independently. A user with 3 terminals open for 2 hours each has 6 hours of coding, but each terminal reports only 2. This is the most commonly cited limitation.

### Design

#### State file naming

Replace single `pace-control-state.json` with per-terminal files:

```
~/.claude/pace-control-state.{PPID}.json
```

`PPID` is the parent PID of the hook script process. In Claude Code's hook execution model, hooks run as direct child processes of the `claude` binary, so `$PPID` is the Claude Code process PID — stable across all hook invocations from the same terminal.

**Verified:** In the current hook execution context, `$PPID` resolves to the `claude` process (confirmed via `ps -o comm= -p $PPID` returning `claude`). If a future Claude Code version changes hook execution to use an intermediate wrapper, `$PPID` would need to be replaced with a process-tree walk (find ancestor with `comm=claude`). For now, `$PPID` is correct and stable.

#### Migration

On first run, if the old `pace-control-state.json` exists:
1. Rename it to `pace-control-state.${PPID}.json`
2. Delete the old file
3. Continue with the renamed file

Migration is idempotent: the old file is checked for existence first; if already gone, nothing happens. Both `session-start.sh` and `session-tracker.sh` include this check, so whichever runs first handles the migration and the other finds nothing to migrate. If `session-tracker.sh` fires before `session-start.sh` (possible depending on hook ordering), it handles migration correctly.

#### Aggregation

On each `session-tracker.sh` invocation, after reading own state:

```bash
# Scan all state files, compute aggregate (excludes own terminal)
OTHER_TERMINALS=0
AGGREGATE_MINUTES=0
for PEER_STATE in "$CLAUDE_DIR"/pace-control-state.*.json; do
  [ -f "$PEER_STATE" ] || continue
  # Skip own state file
  [ "$PEER_STATE" = "$STATE_FILE" ] && continue
  # Extract PID from filename
  PEER_PID=$(echo "$PEER_STATE" | grep -oE '[0-9]+\.json$' | grep -oE '^[0-9]+')
  # Skip if PID is dead (and clean up stale file)
  if ! kill -0 "$PEER_PID" 2>/dev/null; then
    rm -f "$PEER_STATE"
    continue
  fi
  # Skip if peer session has effectively ended (idle > GAP_THRESHOLD)
  PEER_LAST=$(python3 -c "import json; print(json.load(open('$PEER_STATE')).get('lastCheck',0))" 2>/dev/null || echo 0)
  if [ $((NOW - PEER_LAST)) -gt "$GAP_THRESHOLD" ]; then
    continue  # Stale but alive — don't count, don't delete (peer may resume)
  fi
  PEER_MINUTES=$(python3 -c "import json; print(json.load(open('$PEER_STATE')).get('totalMinutes',0))" 2>/dev/null || echo 0)
  OTHER_TERMINALS=$((OTHER_TERMINALS + 1))
  AGGREGATE_MINUTES=$((AGGREGATE_MINUTES + PEER_MINUTES))
done
# AGGREGATE_MINUTES now contains only OTHER terminals' time
# Add own terminal's time for total:
TOTAL_AGGREGATE=$((AGGREGATE_MINUTES + ELAPSED_MINUTES))
```

`OTHER_TERMINALS` counts peers only (skips own `$STATE_FILE`). Display uses `OTHER_TERMINALS` directly — no off-by-one.

#### Surfacing

**Session start (`session-start.sh`):** If `OTHER_TERMINALS > 0`:
```
You have N other Claude Code session(s) running (combined total: Xh Ym across all terminals).
```
One line, inserted into the existing session-start output. Not shown if no other terminals.

**L2+ interventions (`session-tracker.sh`):** If `OTHER_TERMINALS > 0`, append to the intervention:
```
Note: You also have N other terminal(s) running. Combined time across all: Xh Ym.
```

**Micro-nudges:** Do NOT include aggregate in micro-nudges — they must stay one-liners.

#### Stale detection

- `kill -0 $PID 2>/dev/null` — returns 0 if process exists, non-zero if dead
- Dead state files are deleted on detection
- No file locking needed — each terminal writes only its own file. Note: `json.dump` is not truly atomic (no write-to-temp-then-rename), and there is a theoretical race where cleanup deletes a peer's file while it's being written. In practice this is inconsequential: hooks run serially within each terminal, and the worst case is a temporarily missing state file that gets recreated on the next tool call.

#### session-start.sh changes

- `STATE_FILE` becomes `"${CLAUDE_DIR}/pace-control-state.${PPID}.json"`
- Migration logic: if old `pace-control-state.json` exists, rename to PID-stamped
- Aggregation scan added before output
- Multi-terminal stats surfaced in greeting if applicable

#### session-tracker.sh changes

- `STATE_FILE` becomes `"${CLAUDE_DIR}/pace-control-state.${PPID}.json"`
- Migration logic (same as session-start)
- Aggregation scan after reading own state
- L2+ blocks include aggregate stats when `OTHER_TERMINALS > 0`
- Gap detection: when logging completed session to history, clean up own state file
- Default state init uses PID-stamped path

#### test-output.sh changes

- All tests use PID-stamped state files
- New test cases:
  - Multi-terminal: 2 state files exist, aggregation shows combined time
  - Stale detection: state file with dead PID is removed
  - Migration: old `pace-control-state.json` is renamed on first run

### Files

- Modified: `scripts/session-tracker.sh`
- Modified: `scripts/session-start.sh`
- Modified: `scripts/test-output.sh`

## 2. Healthy-Stop Streak

### Problem

No positive reinforcement for stopping at healthy times. The system only intervenes when sessions go too long. Research shows inverted streaks (tracking desired behaviour) leverage loss aversion (2x pain of losing vs gaining) without the downsides of usage-encouraging streaks.

### Design

#### Data model

Extend `pace-control-history.json` session entries with:

```json
{
  "start": 1711234567,
  "end": 1711241767,
  "minutes": 120,
  "prompts": 30,
  "startHour": 14,
  "healthyStop": true
}
```

Add a top-level `streak` field:

```json
{
  "sessions": [...],
  "streak": {
    "current": 5,
    "best": 12,
    "lastUpdated": 1711241767
  }
}
```

#### Healthy stop detection

In `session-tracker.sh` gap detection block, when logging a completed session. **Critical ordering:** the healthy-stop check must read `WIND_DOWN_LEVEL` from the previous session's state (lines 67-70) BEFORE the state reset (lines 110-115) wipes it to 0. The existing Python block that logs sessions (lines 80-106) runs before the reset — the streak logic must be embedded there.

```python
# Determine if the previous session ended healthily
# windDownLevel 0 = never reached L3 = definitely healthy
# windDownLevel 3 = reached L3 but stopped before L4 = healthy (user heeded warning)
# windDownLevel 4 = reached L4 = unhealthy (pushed too far)
healthy = ($WIND_DOWN_LEVEL < 4)
```

Simplified: any session that ends before L4 is healthy. L4 means you pushed past the strong recommendation — that's the line.

Why not L3? Because reaching L3 and then stopping (via Safe-Save or naturally) is exactly the desired behaviour. L3 is the system working as intended.

#### Streak logic

In the Python block that logs sessions to history:

```python
# Update streak
streak = history.get('streak', {'current': 0, 'best': 0, 'lastUpdated': 0})
if healthy:
    streak['current'] += 1
    streak['best'] = max(streak['best'], streak['current'])
else:
    streak['current'] = 0
streak['lastUpdated'] = $NOW
history['streak'] = streak
```

#### Surfacing

**Session start only.** Never during work — zero distraction.

In `session-start.sh`, after the weekly stats computation, read streak data from history:

```python
# Inside the existing Python block that reads history
streak = history.get('streak', {'current': 0, 'best': 0})
streak_current = streak.get('current', 0)
streak_best = streak.get('best', 0)
```

Then surface streak info after the weekly stats line:

If `streak.current >= 2`:
```
Healthy stop streak: N sessions in a row.
```

If streak just broke (current == 0 and best >= 2):
```
Last session ran long. Previous best: N sessions. Fresh start now.
```

One line max. Part of the existing weekly context block. Not shown if `current < 2` (positive) or `best < 2` (broken). This prevents showing "Previous best: 0" for users who never built a streak.

#### Tone

- "Healthy stop streak: 5 sessions" — factual, not celebratory
- "Previous streak: 8" when broken — data, not guilt (Beeminder's "derailing is not failing")
- Never use words like "congratulations" or "great job"

### Files

- Modified: `scripts/session-tracker.sh` (gap detection — add healthyStop to session log, update streak)
- Modified: `scripts/session-start.sh` (add streak display to output)
- Modified: `scripts/test-output.sh` (add streak test cases)

## 3. Show HN Launch Prep

### Problem

The tool exists but has no distribution. Research shows HN front-page averages 289 GitHub stars in one week, and Tuesday-Wednesday launches outperform weekends.

### Design

This is not a code change. Two deliverables:

#### 3a. Show HN post draft — `docs/show-hn-draft.md`

Structure:
- Title: "Show HN: Pace Control – Session health guardrails for Claude Code" (avoid adversarial framing — HN responds better to problem-first than blame-first)
- Body: Problem-first (personal story, Garry Tan's 19-hour admission, the "Addicted to Claude Code–Help" HN thread). Then mechanism (progressive intervention, not a break timer). Then one-command install.
- Links: GIF demo, GitHub repo

#### 3b. Demo simulation script — `scripts/record-demo.sh`

A script that rapidly simulates the full L0→L1→L2→L3→micro-nudge→L4 progression by writing state files and running the tracker. Output suitable for recording with `asciinema rec` or `terminalizer`.

The script:
1. Sets up a clean temp HOME
2. Runs session-start.sh (shows clean start)
3. Simulates L0 (silent — shows nothing)
4. Simulates L1 (gentle awareness)
5. Simulates L2 (evidence nudge)
6. Simulates L3 (full Safe-Save)
7. Simulates L3 micro-nudge (one-liner)
8. Simulates L3 silent (shows nothing — proves it's not spammy)
9. Simulates L4 (mandatory wind-down)
10. Each step has a `sleep 2` and a clear header for readability

### Files

- Created: `docs/show-hn-draft.md`
- Created: `scripts/record-demo.sh`

1. **Multi-terminal** first — larger scope, touches more code, must be stable before streak
2. **Healthy-stop streak** second — builds on the history.json that multi-terminal also touches
3. **Show HN prep** last — marketing assets, no code dependencies
