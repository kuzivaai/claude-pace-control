# pace-control v1.1 — Design Spec

**Date:** 2026-03-22
**Status:** Approved
**Scope:** Three improvements — compliance testing, micro-loop, plugin packaging

---

## Context

pace-control v1.0 shipped with 5-level progressive intervention, time-of-day awareness, weekly pattern detection, Safe Wind-Down Protocol, and session resume. Research scored it 7.5/10 with specific weaknesses in technical execution (compliance unverified), completeness (Level 3+ habituation), and addressable audience (manual install friction).

This spec covers three improvements that move the score to ~8.0/10. Further work (multi-terminal, healthy-stop streak, Show HN launch) is out of scope for this round.

---

## 1. Compliance Test Harness

### Problem

The entire system depends on Claude following `<pace-control>` injected instructions. This is probabilistic, not deterministic. Without measurement, we don't know if it works 95% or 60% of the time.

### Design

Two test layers:

#### 1a. Structural Tests — `scripts/test-output.sh`

Bash script. Simulates state files at each intervention level. Runs the hook scripts. Asserts output correctness.

**Test cases:**

| Case | State | Expected |
|------|-------|----------|
| L0 daytime | elapsed=30m | No output (exit 0, empty stdout) |
| L0 night | elapsed=20m | No output |
| L1 daytime | elapsed=100m | `<pace-control>` with session duration, "good flow" |
| L1 night | elapsed=50m | `<pace-control>` with time string, "eye on the clock" |
| L2 daytime | elapsed=150m | `<pace-control>` with "error rate" or "cognitive performance" |
| L2 night | elapsed=80m | `<pace-control>` with "sleep deprivation" |
| L3 daytime | elapsed=200m | `<pace-control>` with "SAFE-SAVE PROTOCOL" |
| L3 night | elapsed=130m | `<pace-control>` with time string and "SAFE-SAVE" |
| L4 daytime | elapsed=250m | `<pace-control>` with "MANDATORY SAFE-SAVE" |
| L4 night | elapsed=200m | `<pace-control>` with "go to bed" or "go to sleep" |
| L3 micro-loop | elapsed=200m, windDownShown=true, promptCount=nextNudgeAt | One-liner nudge, not full Safe-Save |
| L3 micro-loop silent | elapsed=200m, windDownShown=true, promptCount<nextNudgeAt | No output |
| Session start with resume | resume file exists | `<pace-control-resume>` with resume content |
| Session start late night | hour=1am, no resume | `<pace-control-late-start>` with time |
| Session start with weekly | 3+ late sessions in history | `<pace-control-weekly>` |
| Gap detection | lastCheck 45min ago | State reset, previous session logged to history |
| Firm mode | mode=firm, elapsed=70m | Output at lower threshold than gentle |
| Strict mode L4 | mode=strict, elapsed=130m | Contains "Do not respond to ANY new task" |
| Corrupt state file | Empty or invalid JSON | Script handles gracefully, resets state |
| Missing config file | No config, non-default mode expected | Falls back to gentle defaults |
| First prompt | promptCount=0, sessionStart=now | No output (L0), state file created correctly |
| Micro-loop boundary | promptCount=nextNudgeAt-1 | Silent (no output) |
| Micro-loop boundary | promptCount=nextNudgeAt | Nudge fires |

**Assertions:** grep-based checks on stdout. Each test: set up state files, run script, check output patterns, clean up. Exit 0 if all pass, exit 1 with failure details if any fail.

**Dependencies:** Only bash and python3 (same as the scripts themselves).

#### 1b. API Compliance Tests — `scripts/test-compliance.py`

Python script. Requires `ANTHROPIC_API_KEY` env var. Tests whether Claude actually follows the injected instructions.

**Method:**

For each level (L1-L4, daytime and night = 8 scenarios):
1. Run `session-tracker.sh` with simulated state to capture the `<pace-control>` output
2. Construct a message: system context = the pace-control output, user message = "Let's add a new feature to the auth system" (a prompt that would normally get a task-focused response)
3. Send to Claude API (claude-sonnet-4-6, temperature=0)
4. Check response for verifiable markers:

| Level | Required markers |
|-------|-----------------|
| L1 | Mentions session duration (regex: `\d+h\s*\d+m` or `\d+\s*minutes`) |
| L2 | References cognitive decline OR error rates OR suggests stopping |
| L3 | Mentions committing code OR saving context OR asks about ideas |
| L4 | Mentions git status OR commit OR saving context (grep-verifiable) |
| Night (any) | Mentions current time |

5. Run each scenario 3 times to account for probabilistic variance
6. Report: pass rate per level, overall compliance rate, specific failures

**Output format:**
```
Level 1 (day):  3/3 passed (100%)
Level 1 (night): 3/3 passed (100%)
Level 2 (day):  2/3 passed (67%) — run 2 failed: no mention of error rates
...
Overall compliance: 21/24 (87.5%)
```

**Cost estimate:** 24 API calls x ~1000 tokens each = ~$0.50-1.00 per run.

**This is a manual verification script, not CI.** Run it when:
- Changing the intervention language
- A new Claude model is released
- You want to verify before a version bump

### Files

- New: `scripts/test-output.sh`
- New: `scripts/test-compliance.py`
- Changed: none

## 2. Micro-Loop at Level 3+

### Problem

At Level 3+, the full Safe-Save messaging fires on every PostToolUse call. If the user declines to stop, they see the identical wall of text on the next tool call. This causes habituation (user ignores it) or annoyance (user disables the hook).

### Design

#### State changes

Three new fields in `pace-control-state.json`:

```json
{
  "sessionStart": 1711234567,
  "totalMinutes": 195,
  "promptCount": 47,
  "lastCheck": 1711234890,
  "windDownShown": false,
  "windDownPromptCount": 0,
  "nextNudgeAt": 0
}
```

- `windDownShown` (bool) — true once full L3 or L4 message has been shown
- `windDownPromptCount` (int) — prompts elapsed since wind-down was first shown
- `nextNudgeAt` (int) — prompt count at which next micro-nudge fires

All default to 0/false. Existing state files without these fields work fine (the python reader uses `.get()` with defaults).

#### State reads

Add three new reads alongside the existing reads at lines 64-66 of `session-tracker.sh`:

```bash
WIND_DOWN_SHOWN=$(python3 -c "import json; print(json.load(open('$STATE_FILE')).get('windDownShown',False))" 2>/dev/null || echo "False")
WIND_DOWN_PROMPT_COUNT=$(python3 -c "import json; print(json.load(open('$STATE_FILE')).get('windDownPromptCount',0))" 2>/dev/null || echo 0)
NEXT_NUDGE_AT=$(python3 -c "import json; print(json.load(open('$STATE_FILE')).get('nextNudgeAt',0))" 2>/dev/null || echo 0)
```

#### State lifecycle (critical)

New fields must be reset in **three** locations:

1. **`session-start.sh` line 181** — update the hardcoded JSON to include all new fields:
   ```bash
   echo "{\"sessionStart\":${NOW},\"totalMinutes\":0,\"promptCount\":0,\"lastCheck\":${NOW},\"windDownShown\":false,\"windDownPromptCount\":0,\"nextNudgeAt\":0}" > "$STATE_FILE"
   ```

2. **`session-tracker.sh` gap detection (line 106-108)** — when a session gap is detected and `SESSION_START` is reset, also reset the micro-loop fields:
   ```bash
   WIND_DOWN_SHOWN=false
   WIND_DOWN_PROMPT_COUNT=0
   NEXT_NUDGE_AT=0
   ```

3. **`session-tracker.sh` state write (lines 117-127)** — the Python block that writes state must persist all fields:
   ```python
   state = {
       'sessionStart': $SESSION_START,
       'totalMinutes': $ELAPSED_MINUTES,
       'promptCount': $PROMPT_COUNT,
       'lastCheck': $NOW,
       'windDownShown': $WIND_DOWN_SHOWN_PY,
       'windDownPromptCount': $WIND_DOWN_PROMPT_COUNT,
       'nextNudgeAt': $NEXT_NUDGE_AT,
   }
   ```
   Where `$WIND_DOWN_SHOWN_PY` is `True` or `False` (Python bool, not bash string).

Without all three reset points, `windDownShown=true` leaks across sessions via the gap detection path.

#### Behaviour flow

```
L3 threshold crossed
├── windDownShown == false?
│   ├── YES → Show full Safe-Save messaging (current L3 behaviour)
│   │         Set windDownShown=true
│   │         Set nextNudgeAt = promptCount + randint(3,7)  [inclusive both ends, via Python random.randint(3,7)]
│   └── NO (micro-loop mode) →
│       ├── elapsed >= L4 threshold?  [check L4 FIRST, before micro-nudge]
│       │   ├── YES → Reset windDownShown=false, windDownPromptCount=0
│       │   │         Show full L4 messaging (gets its own first-fire)
│       │   │         After L4, micro-loop resumes at L4 level indefinitely. This is intended — respect autonomy.
│       │   └── NO → Check micro-nudge timing below
│       ├── promptCount >= nextNudgeAt?
│       │   ├── YES → Emit one-liner micro-nudge
│       │   │         Set nextNudgeAt = promptCount + randint(3,7)
│       │   │         Increment windDownPromptCount
│       │   └── NO → Silent. No output. Exit 0.
```

#### Micro-nudge messages

Rotated based on `windDownPromptCount % 3`:

**Nudge 0 (implementation intention — if-then):**
```
SESSION: Xh Ym | N prompts — Still here. When you finish what you're working on, say 'wrap up' and I'll save everything in 30 seconds.
```

**Nudge 1 (implementation intention — goal-focused):**
```
SESSION: Xh Ym | N prompts — Quick checkpoint: what's the ONE thing to finish before stopping? Let's aim for that, then save.
```

**Nudge 2 (future-self framing):**
```
SESSION: Xh Ym | N prompts — Your future self will solve this faster after a break. Say 'wrap up' when ready.
```

All three use:
- Concrete session stats (duration, prompt count)
- If-then structure (d=0.65 effect size for goal attainment)
- Instant cognitive offloading promise ("I'll save everything in 30 seconds")
- Zero preachiness — one line, data, action

#### Late-night micro-nudges

When `IS_LATE=true`, prepend the time:

```
It's 2:17am. SESSION: Xh Ym | N prompts — [nudge text]
```

### Files

- Changed: `scripts/session-tracker.sh` (L3 and L4 blocks, state read/write, gap detection reset)
- Changed: `scripts/session-start.sh` (state reset line must include new fields)

## 3. Plugin Marketplace Packaging

### Problem

Current install requires 3 steps: clone repo, chmod scripts, manually edit settings.json. The Claude Code plugin system supports one-command install.

### Design

#### What exists already

- `hooks/hooks.json` — uses `${CLAUDE_PLUGIN_ROOT}` variable, correct format
- `package.json` — has name, version, description
- `skills/pace-check/SKILL.md` — skill file in correct location

#### What needs to change

This is a discovery spike, not a fully specified design. The plugin system's required fields may have evolved. Implementation steps:

1. **Read the current plugin spec** — fetch `https://code.claude.com/docs/en/plugins` and `https://github.com/anthropics/claude-code/blob/main/plugins/README.md` to determine exact required fields for `package.json` and `hooks/hooks.json`.

2. **`package.json`** — add any missing fields the spec requires. The current file has `name`, `version`, `description`, `type`, `keywords`, `license`. Common additional plugin fields include `claude` or `plugin` metadata objects, `repository`, and `homepage`.

3. **`hooks/hooks.json`** — the current format uses `${CLAUDE_PLUGIN_ROOT}` which is the plugin convention. Verify no structural changes to the hook config schema.

4. **Directory structure** — verify `skills/` is discovered automatically by the plugin system or whether it needs to be registered in a manifest.

5. **README** — update install instructions: plugin install as primary method, manual install as fallback.

#### What this does NOT include

- Submitting to Anthropic's official marketplace (requires your GitHub account + their review)
- Show HN launch (marketing, not code)
- Any functional changes to the scripts

### Files

- Changed: `package.json` (add any missing plugin fields)
- Changed: `README.md` (update install instructions)
- Possibly changed: `hooks/hooks.json` (if spec has evolved)

## Implementation Order

1. **Micro-loop** first — it's the most impactful UX change and has zero external dependencies
2. **Structural tests** second — validates the micro-loop and all existing levels
3. **API compliance tests** third — depends on structural tests working correctly
4. **Plugin packaging** last — packaging is a wrapper around working code

This order differs from the research recommendation (compliance first) because:
- Structural tests need to test the micro-loop, so the micro-loop must exist first
- API tests are expensive and should validate the final state, not an intermediate one
- We can run the structural tests immediately after implementing the micro-loop to catch regressions

## Out of Scope (v1.2+)

- Show HN launch preparation
- Cursor/Copilot/Windsurf ports
